# Project Overview

This repository provides tools and resources for customizing and building a Linux kernel with a focus on packet filtering using the Netfilter subsystem. It also includes scripts and configurations for setting up a testing environment within virtual machines.

---
## The project of Sangesh is the M10 tcp flag counter 
The project carries the backbone of the Kernal-Playground.
The part that was modified to make this work is the file snf_lkm.c which is located in the Kernal/modules folder

## Goal of this project
the goal of this project is to count how many tcp flags entered and leave the package
 syn_count = 0;
 ack_count = 0;
 fin_count = 0;
 rst_count = 0;
## How it works
Network Interface
        ↓
Netfilter Hook (PRE_ROUTING)
        ↓
Check IPv4 Packet
        ↓
Check TCP Header
        ↓
Read TCP Flags
        ↓
Update Counters
        ↓
Pass Packet Normally

The module registers a hook with the Linux Netfilter framework. Netfilter allows kernel modules to inspect network packets as they pass through the Linux networking stack.
When a packet reaches the selected Netfilter hook:
The module verifies that the packet is an IPv4 packet.
It checks whether the transport protocol is TCP.
The TCP header is extracted.
The TCP flags field is inspected.
The corresponding counters are incremented.
The module then allows the packet to continue its normal path through the network stack.
cimal value is looked at when a packet enters 

syn_flag = 0x02;
ack_flag = 0x10;
fin_flag = 0x01;
rst_flag = 0x04;
if it sees this specific hexadecimal it adds +1 to the counter accordingly

# Testing Methodology

The TCP Flag Counter module was tested inside the Kernel Playground virtual machine environment.

## 1. Loading the Module

The kernel module was compiled and loaded into the kernel using:

```bash
sudo insmod snf_lkm.ko
```

The kernel log was checked to verify that the module was successfully registered with Netfilter:

```bash
dmesg | tail
```

## 2. Generating TCP Traffic

Different network activities were performed to generate TCP packets containing various flags.

### SYN and ACK Flags

A TCP connection was initiated using:

```bash
curl https://www.google.com
```

or

```bash
ping google.com
```

The TCP three-way handshake generates SYN and ACK packets which should increase the corresponding counters.

### FIN Flags

TCP connections were closed normally after communication ended. During connection termination, FIN packets were generated and counted by the module.

### RST Flags

TCP reset packets were generated by attempting connections to closed ports or by abruptly terminating active TCP connections. These packets contain the RST flag and should increment the reset counter.

## 3. Unloading the Module

After sufficient traffic had been generated, the module was removed:

```bash
sudo rmmod snf_lkm
```

When the module was unloaded, the final values of the counters were printed to the kernel log.

## 4. Verifying Results

The output was inspected using:

```bash
dmesg | tail -50
```

Example output:

```text
TCP Flag Statistics
-------------------
SYN Count : 124
ACK Count : 356
FIN Count : 118
RST Count : 7
```

The counters increased as expected when the corresponding TCP flags were observed in network traffic, confirming the correct operation of the TCP Flag Counter module.





## Repository Structure

### 1. `kernel` Folder
Contains the kernel configuration, source code, and a custom kernel module. The `Makefile` in this folder supports various tasks:

- **config**: Copies the kernel configuration file into the Linux kernel source directory.
- **kbuild**: Builds the entire Linux kernel.
- **install**: Creates a symbolic link to the compiled kernel (`bzImage`) inside the `tests/vm` folder, enabling the VM to boot with this custom kernel.
- **kmodule**: Builds the custom kernel module and copies the resulting `.ko` file into the `tests/vm/shared` folder. This allows the VM's guest OS to load the module using `insmod`.

**Note:** The `modules` subfolder contains the source code for the custom kernel module and a `Makefile` to compile and install it into the shared VM folder. Refer to the specific `README.md` within the `modules` folder for instructions on recompiling and copying the module after modifications.

---

### 2. `podman` Folder
Includes scripts and configurations to set up a containerized environment for building and running the kernel and modules.

**Note:** For setup instructions, see the `README.md` inside the `podman` folder.

---

### 3. `tests` Folder
Contains:
- `vm`: The root filesystem used by the virtual machine to run the guest OS with the custom kernel. It also includes scripts to build, run, and connect to the VM.
- `scripts`: Collection of scripts for testing and various use cases within the VM environment.
